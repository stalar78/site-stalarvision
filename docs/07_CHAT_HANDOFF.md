# Chat Handoff

## Что это за файл

Этот файл нужен как короткая handoff-сводка для следующего чата с Codex/Copilot.

Его задача:
- быстро напомнить текущее фактическое состояние проекта;
- сократить риск повторного анализа уже принятых решений;
- зафиксировать, какие вещи уже считаются рабочими, а какие всё ещё требуют подтверждения владельца.

## Текущее фактическое состояние

На текущий момент в проекте уже есть:
- опубликованный сайт на production-домене `https://stalarvision.ru`;
- настроенный HTTPS / SSL;
- главная страница, privacy/legal page и 404;
- реальные контакты владельца: email, Telegram, GitHub и телефон;
- рабочая форма обращения через Web3Forms;
- базовый legal/privacy слой, согласованный с реальной работой формы;
- post-launch SEO/share polish:
  - canonical URL,
  - Open Graph meta,
  - Twitter meta,
  - `robots.txt`,
  - `sitemap.xml`;
- уточнённая верхнеуровневая подача под три сценария: новый проект, доработка существующего решения, аудит / технический разбор;
- отполированные services, contact flow, representative examples и FAQ под эту же рамку без редизайна.

Проект находится на стадии:
**рабочая публичная версия + точечная post-launch полировка**

## Главные source of truth файлы

- `src/data/profile.ts`
  Хранит бренд, владельца, юридические данные, контакты, CTA и базовые SEO-данные сайта.
- `src/data/site.ts`
  Хранит тексты и структуру основных homepage-секций.
- `src/data/services.ts`
  Хранит секцию услуг и формулировки по аудиту / техразбору.
- `src/data/cases.ts`
  Хранит честные representative-примеры формата проектов.
- `src/data/faq.ts`
  Хранит FAQ и короткие ответы на типовые вопросы перед обращением.
- `src/data/contacts.ts`
  Хранит contact-секцию, контакты для UI и настройки формы Web3Forms.
- `src/data/legal.ts`
  Хранит privacy/legal copy и реквизиты для `/privacy`.
- `src/lib/meta.ts`
  Применяет runtime meta для страниц.
- `public/robots.txt`
  Текущие robots-правила для production-сайта.
- `public/sitemap.xml`
  Текущая sitemap для production-маршрутов.

## Что уже не нужно трактовать как черновик

Следующие вещи уже являются реальными решениями проекта:
- сайт опубликован и не является просто локальным шаблоном;
- контакты в интерфейсе реальные, а не временные placeholder-значения;
- форма на сайте рабочая и отправляет обращения через Web3Forms;
- privacy/legal тексты уже должны оставаться согласованными с реальной работой формы;
- SEO/share слой уже есть и должен поддерживаться, а не собираться с нуля заново;
- hero / services / contact / representative examples / FAQ уже сознательно выровнены под сценарии: новый проект / доработка / аудит / техразбор.

## Что нельзя ломать следующими правками

Не ломать без явной причины:
- честную подачу сайта как сайта одного независимого разработчика;
- отсутствие фейковых отзывов, метрик, клиентов и кейсов;
- централизованный source of truth в `src/data/*`;
- связку `profile.ts` -> meta -> `robots.txt` / `sitemap.xml`, если меняется production domain или OG image;
- privacy/legal формулировки, связанные с формой и контактами.

## Что всё ещё требует подтверждения владельца

Пока не считается подтверждённым фактом:
- реальные публичные кейсы или согласованные анонимизированные кейсы;
- реальные отзывы, если они вообще понадобятся на сайте;
- точные рамки услуги аудита / технического разбора / базовой проверки безопасности, если эти формулировки захочется усиливать;
- необходимость более формального legal-слоя сверх текущего базового privacy/legal уровня;
- финальная OG-картинка, если владелец решит заменить текущий локальный asset;
- любые новые интеграции аналитики, cookies или внешних сервисов.

## Хороший формат следующего шага

Если проект продолжается дальше, следующий запрос лучше формулировать так:
1. указать конкретную секцию или слой (`content`, `mobile polish`, `SEO`, `legal`, `contact flow`);
2. перечислить только реальные подтверждённые данные;
3. явно запретить выдумывать кейсы, цифры, отзывы и обещания;
4. попросить точечные изменения без редизайна и без большого рефакторинга.
