# Codex Workflow

## Роли

### GPT
GPT — архитектор и руководитель проекта.

Задачи GPT:
- анализировать текущее состояние проекта;
- определять следующий логичный шаг;
- писать точный промпт для Codex/Copilot;
- делать ревью результата;
- следить за целостностью проекта;
- следить за актуальностью docs;
- обновлять проектную документацию непосредственно в GitHub после подтверждённой реализации.

### Codex / Copilot
Codex и Copilot — исполнители изменений в коде.

Их задача:
- вносить изменения по точному промпту;
- не переизобретать проект;
- не отклоняться в большие лишние переделки;
- не придумывать бизнес-факты;
- запускать предусмотренные проектом проверки, прежде всего `npm run build`;
- перечислять изменённые файлы и сообщать результат проверки.

## Формат работы

Базовый цикл:
1. GPT анализирует проект и текущее состояние.
2. GPT определяет следующий логичный шаг.
3. GPT пишет готовый промпт для Codex/Copilot на английском языке.
4. Codex/Copilot изменяет только код и связанные с задачей статические файлы.
5. Codex/Copilot запускает `npm run build` и сообщает результат.
6. Пользователь проверяет список изменений, создаёт commit и отправляет код в GitHub.
7. GPT читает актуальный код из GitHub и проводит ревью.
8. GPT самостоятельно обновляет нужные файлы в `docs/` непосредственно в GitHub.
9. Пользователь выполняет локальный `git pull --ff-only origin main`.
10. После успешного ревью выполняется безопасный production-деплой.

## Разделение ответственности по файлам

### Codex / Copilot может изменять
- `src/**`;
- `public/**`, если это прямо требуется задачей;
- конфигурационные файлы проекта, если это прямо требуется задачей.

### Codex / Copilot не изменяет
- любые файлы в `docs/`;
- документацию проекта в других местах без отдельного указания;
- серверную конфигурацию;
- файлы и каталоги других проектов на production-сервере.

Документацию обновляет GPT отдельно после ревью фактически реализованного состояния.

## Git-правила

Codex / Copilot не должен:
- создавать commit;
- выполнять `git push`;
- отправлять изменения в `main`;
- выполнять merge или rebase;
- изменять историю репозитория.

Commit и push выполняет владелец проекта после проверки списка файлов.

Предпочтительный порядок:

```bash
git status
git add <явный список файлов>
git commit -m "Краткое описание изменения"
git push origin main
```

После того как GPT обновил документацию в GitHub:

```bash
git pull --ff-only origin main
```

## Важные правила для промптов

- Все готовые промпты для Codex/Copilot писать на английском языке.
- Всё общение с пользователем, объяснения, анализ, review и стратегические решения писать на русском языке.
- Названия файлов, путей, компонентов, переменных и технических сущностей сохранять в той форме, в которой они используются в проекте.
- В каждом промпте явно указывать: `Не изменять никакие файлы в docs/`.
- В каждом промпте явно указывать: `Не создавать commit и не выполнять push`.
- Если для корректного ревью не хватает контекста, GPT должен прямо сказать, каких файлов не хватает и зачем.
- Основным источником для ревью после push является актуальное состояние репозитория GitHub.

## Что запрещено

Нельзя:
- хаотично менять архитектуру;
- придумывать бизнес-контент;
- терять continuity проекта;
- отвечать так, будто проект надо делать с нуля, если это не так;
- смешивать кодовую задачу Codex с обновлением project docs;
- выполнять серверные изменения без отдельного точного плана и проверки области воздействия.
